# 推理重叠验证实验 — 结论摘要

## 研究问题

**代码生成能力是否是大型语言模型文字推理能力的来源？**

具体来说：模型生成代码时需要将问题分解为步骤序列（循环、递归、条件分支），这些"拆解模式"在 SAE 特征空间中能否被观测到迁移到文字推理任务中？

---

## 实验设计

使用 **Qwen3-8B + Sparse Autoencoder (SAE, 65536 维稀疏空间, top-100 激活)**，通过三类实验递进验证：

### Prompt 分组

| 组 | 内容 | 数量 | 角色 |
|---|---|---|---|
| **A (Code)** | 代码生成任务 | 30 | 提供"代码拆解特征" |
| **D (Description)** | 同概念纯事实描述（控制共享概念） | 15 | **关键控制组** |
| **C (Reasoning-code)** | 代码相关推理（Fibonacci、stack、排序等） | 15 | 实验组 |
| **C' (Reasoning-pure)** | 纯推理（行程问题、年龄、概率等） | 15 | 对照组 |

### 候选集设计

```
A-D = 代码独有特征（排除概念共享后的"拆解过程特征"）→ 假说关心的
A∩D = 概念共享特征（代码和描述都涉及的概念）→ 语义干扰
```

### 三阶段验证

| 阶段 | 内容 |
|---|---|
| **1. 相关性分析** | 扫描所有 prompt，统计推理任务中 A-D 和 A∩D 特征的出现频率 |
| **2. 概念共享控制** | 验证 D 控制组自身在 A-D 上的激活量（应接近 0） |
| **3. 因果干预** | Negate 三种特征（A-D / A∩D / D-A），观测推理是否退化 |

---

## 核心发现

### 阶段 1-2：相关性分析

排除概念共享后（V2 修正版），**代码独有特征（A-D）在推理中不占优势**。概念共享特征（A∩D）的数量看起来更多，但后续验证发现 A∩D 是通用语言能力特征而非领域概念特征。

### 阶段 3：因果干预（决定性证据）

在 Layer 31（最后一层）上，以相同剂量 negate 三类特征：

| 干预对象 | 剂量 | 推理结果 | 对照组结果 | 结论 |
|---|---|---|---|---|
| **A-D（代码独有）** | top-10 → **全部 87 个** | **正确** | — | **代码特征在推理中无因果作用** |
| **A∩D（通用语言）** | top-10 | **全面崩溃** | — | 这是文本生成的基础组件，非概念特征 |
| **D-A（描述独有，控制）** | top-3 | 正确 | 正常 | 对照组验证通过 |

**全量 negate 87 个代码专用特征后，Bubble sort、Stack LIFO、Fibonacci 推理全部保持正确，与 baseline 无差异。**

### Hard Prompt 验证 — 排除"背答案"干扰

以上推理任务可能过于简单，模型可能通过记忆而非真正的推理来回答。为排除这一干扰，我们生成了 **Hard Reasoning Prompts**（`surgenry/data/prompts/hard_reasoning_code.json`），每类推理都带随机参数，确保模型无法通过记忆解答：

| 类型 | 随机化方式 | 确保无法记忆 |
|---|---|---|
| Stack LIFO | 随机 push 数字、随机序列长度 | 每次参数不同 |
| Insertion Sort | 随机数组、随机插入位置 | 不可能见过完全相同的问题 |
| Selection Sort | 随机数组、随机 pass 次数 | 同上 |
| Sequence Induction | 随机种子数 + 6 种不同的递推规则 | 规则和种子都随机，无法匹配训练数据 |
| Graph 最短路 | 随机拓扑、随机边权重 | 每张图都是全新的 |
| Water Pouring | 随机桶容量、随机目标值 | 每道题参数不同 |

在 Layer 31 全量 negate 87 个 A-D 特征后测试：

| Hard Reasoning 类型 | 干预结果 | 基线对比 |
|---|---|---|
| Stack LIFO (随机) | **正确** | 一致 |
| Insertion Sort | **正确** | 一致 |
| Selection Sort | **正确** | 一致 |
| Sequence Induction (8,17,35,71→143) | **正确** | 一致 |
| Graph Shortest Path | **正确** | 一致 |
| Water Pouring (11/4→8L) | **正确** | 一致 |

即使使用不可能通过记忆回答的随机参数推理题，全部 negate 代码特征后模型推理仍不受影响。**这排除了"因为任务太简单所以没测到"的质疑。**

### 附加验证：A-D 特征真的是"代码独有"吗？

有人提出：A-D 特征可能不是"代码独有"，而是"广义过程拆解特征"（菜谱步骤、组装说明等非代码场景也有）。为验证这一点，我们扫描了 11 个非代码过程拆解任务（菜谱、组装家具、滴定实验、换轮胎等），计算它们与 A-D 特征的重叠：

| 层 | A-D 总数 | E(过程)中激活的 A-D | 比例 |
|---|---|---|---|
| **Layer 15** | 89 | **17** | **19.1%** |
| **Layer 24** | 85 | **12** | **14.1%** |
| **Layer 31** | 87 | **11** | **12.6%** |

A-D 特征中 **12-19% 也在非代码过程拆解任务中激活**。这意味着 A-D 更准确的标签应该是："主要在代码中出现，但也部分覆盖通用过程拆解的特征"。

但这对主结论无影响：**无论叫"代码独有"还是"广义过程拆解"，negate 这些特征后推理不退化，这是不争的事实。**

跨层干预（三层 261 个）和浅层干预（Layer 15 89 个）虽然导致崩溃，但 **同剂量的 D-A 控制组也一样崩**——说明是总干扰量超过残差流容错阈值，而非代码特征的特异性效应。

---

## 最终结论

**假说"代码拆解为推理提供了 token 桥梁" — 不成立。**

```

代码训练数据         文字推理能力
  │                       │
  ├─ 概念知识 ✓           ├─ 语言因果结构描述（推测来源）
  │   (Fibonacci, 递归)    │   ("如果…那么…""先…后…")
  │                       │
  └─ 拆解过程 ✗           └─ 不从代码迁移
      (SAE 特征无因果作用)    
```

### 具体含义

1. **代码训练对推理的帮助在概念知识层面**（"什么是递归""什么是排序"），而非拆解过程迁移
2. **推理能力不存在代码→文字的迁移路径**。即使使用随机参数（无法记忆的 hard prompts），negate 全部 87 个代码特征后推理仍不受影响
3. **SFT 增强推理应直接使用因果推理数据**，比通过代码增强更有效
4. **文字推理能力更可能来自语言本身的因果结构描述**（"如果…那么…""导致/因为/所以"等语料中的逻辑关系）
5. **"涌现"说在此不成立**：本实验不支持"推理从代码能力涌现"的假设，更接近从通用语言能力中逐步分化

### 方法学教训

- SAE 特征比较必须控制概念域（不相关领域做基线会产生假阳性）
- 两个 prompt 集的交集不一定是"概念共享特征"，更可能是通用高频特征
- 跨层干预时总剂量可能超阈值，控制组失效；干预验证应优先选择深层
- 相关性 ≠ 因果，必须做干预验证
