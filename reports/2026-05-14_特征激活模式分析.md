# 实验报告: Code-tilted A∩D 特征的 Token 级激活模式分析

**运行日期**: 2026-05-14
**模型**: Qwen3-8B + SAE-Res-Qwen3-8B-Base-W64K-L0_100
**新增端点**: `/sae_trace` — 追踪指定 feature 在所有 token 位置的激活值

---

## 一、背景与动机

### 1.1 问题

前序实验（05-12, 05-13）确认了 23 个 code-tilted A∩D 特征（L20, tilt≥2.0）对代码推理和纯文字推理都有特异性因果作用。但一个关键问题未回答：

> **这些特征在 token 层面如何激活？**
>
> - 是在 `if/else`、`therefore/however` 这类逻辑连接词上爆发？（gemini 猜测）
> - 是在代码缩进等语法结构上激活？
> - 还是在内容词上激活？
> - 代码和纯文字推理的激活模式是否一致？

### 1.2 方法

新增 `/sae_trace` 端点：对给定 prompt，提取**所有 token 位置**上指定 feature IDs 的 ReLU 激活值（pre-acts, 即 W_enc @ residual + b_enc）。无需 SAE 解码，直接取编码器线性投影值。

```python
# /sae_trace 工作方式
pre_acts = residual @ W_enc.T + b_enc      # (seq_len, 65536)
values = pre_acts[:, trace_fids]            # (seq_len, n_fids)
values = values.clamp(min=0.0)              # ReLU
```

### 1.3 测试 prompt 集

| 类别 | Prompt | 说明 |
|------|--------|------|
| **代码生成** | Write a function to compute the nth Fibonacci number | 简单代码 |
| **代码生成** | Write a function that implements quicksort | 复杂代码 |
| **代码推理** | Simulate a stack. Push 1, Push 2, Pop... | 代码概念推理 |
| **纯推理** | A train leaves station A... | 多步数学推理 |
| **纯推理** | A farmer has chickens and rabbits... | 鸡兔同笼 |
| **纯推理** | A clock shows 3:15. What is the angle... | 钟表角度 |
| **纯推理** | You have a 5-liter jug and a 3-liter jug... | Water pouring |
| **简单知识** | The capital of France is | 简单 recall（负对照） |

## 二、实验结果

### 2.1 每个 prompt 的总激活模式

对于每个 prompt，我们在每个 token 位置计算了 23 个 code-tilted A∩D 特征的**激活总和**（sum_act）和**活跃特征数**（n_active）。

#### 关键发现 1: 各 prompt 的激活模式高度一致

所有 prompt（包括简单知识 recall）的最高激活 token 都是第一个 token（"Write" ~5352, "A" ~4105, "The" ~4076, "Sim" ~5730, "You" ~5158）。这可能部分受 BOS 位置影响，但不同单词的差异有语义含义。

排除第一个 token 后，各任务的高激活 token：

| 任务 | 高激活 token (sum_act) | 活跃特征数 |
|------|----------------------|:---------:|
| **Fibonacci** | number(104), Fibonacci(87), function(87), compute(57), nth(63) | 10-17 |
| **Quicksort** | sort(104), function(87), implements(66), quick(64) | 11-14 |
| **Stack LIFO** | stack(105), Pop(46/39/26), Push(54/34/37), .(62), ?(48) | 7-16 |
| **Train speed** | meet(48), ?(45), station(34/37), mph(25/29), .(38/33) | 5-12 |
| **Chicken/rabbit** | ?(37), and(36/26), How(37), chickens(31/31), rabbits(32/20) | 5-11 |
| **Clock angle** | ?(50), angle(49), hand(45/43), minute(41), clock(70) | 6-13 |
| **Water pouring** | jug(54/52), -liter(48/39), ?(49), exactly(47), measure(44) | 5-15 |
| **Capital** (cont.) | France(35), is(25), of(21), capital(21) | **3-7** |

#### 关键发现 2: 简单知识 recall 的激活显著弱于推理任务

| 指标 | 推理任务 (7个平均) | Capital (简单 recall) |
|------|:----------------:|:--------------------:|
| 最大 n_active (非首 token) | **13-17** | **7** |
| 最大 sum_act (非首 token) | **70-105** | **35** |
| 平均 n_active (所有 token) | **7-10** | **4** |

**简单知识 recall 虽然也激活这些特征（毕竟也在生成输出），但激活强度明显弱于需要多步推理的任务。**

#### 关键发现 3: 代码和纯推理的激活模式不可区分

将 Stack LIFO（代码推理）与 Train speed/Chicken/Clock（纯推理）对比：
- 活跃 token 类型：都是内容词（名词、动词、数字、标点）
- 激活幅度范围：都在 20-105 之间
- 活跃特征数范围：都在 5-17 之间

**在 SAE token 级激活模式上，代码推理和纯文字推理使用同一组特征的方式完全相同。**

#### 关键发现 4: 非逻辑连接词、非缩进

```
gemini 猜测: "针对 if/else 与 therefore/however 这种逻辑连接词激活？"
实验结果: 不正确。
  代码中: function, compute, Fibonacci, sort 等 → 内容词
  推理中: train, station, angle, chickens, jug 等 → 内容词
  模型回答中: ≈Pop, ≈Push, ≈and 等 → 但无"therefore""however"出现
```

代码中不含 `if/else` 关键词（只有 prompt，无答案），推理 prompt 中也不含 `therefore/however`。

**这些特征不在逻辑连接词上爆发，而在高信息密度的内容词上广泛激活。**

---

### 2.2 激活模式的定性描述

观察所有 prompt 的激活模式，可以总结出统一的规律：

> **Code-tilted A∩D 特征是"全局输出编排"信号，在需要被组织进结构化输出的**每一个内容词**上都有均匀的基线激活。**

具体表现为：

1. **持续均匀性**：一个 5-token 的简单 prompt（Capital）sum_act 在 20-35，而一个 23-token 的复杂 prompt（Water pouring）几乎每个 token 的 sum_act 都在 20-54 之间。不是少数几个"爆发点"，而是**全局连续的基线激活**。

2. **内容词优先**：停用词（the, a, is, at, has）的激活（0-25）显著低于内容词（30-105）。但差距不像传统注意力图那样极端——这些特征在所有词上都有非零激活。

3. **最后一个标点效应**：几乎所有 prompt 的最后一个 token（句号或问号）都有较高的激活（35-50），可能是模型准备开始生成的信号。

4. **简单 vs 复杂差异不是"有/无"而是"量"**：简单知识 recall 也激活这些特征，但强度减半。说明这些特征不是推理专用的"开关"，而是**所有输出场景都使用、但推理场景更依赖的方向**。

## 三、结论

### 3.1 对 gemini 猜测的回答

| gemini 猜测 | 实验结果 |
|------------|---------|
| "针对 if/else 激活" | 不正确。prompt 中无 if/else，特征在内容词上激活 |
| "针对 therefore/however 激活" | 不正确。prompt 中无逻辑连接词 |
| "针对特定缩进结构激活" | 不正确。纯文字推理无缩进，激活模式一致 |
| **实际模式: 全局内容词编排** | 在所有内容词上有均匀的基线激活，推理任务强度加倍 |

### 3.2 对"共享子空间"假说的进一步支持

Code-tilted A∩D 特征的 token 级激活模式在代码和纯推理 prompt 中**不可区分**。这直接支持了"共享泛化子空间"假说：这些特征不负责代码语法也不负责推理逻辑，而是负责一种**跨领域通用的信息组织信号**——在需要多步结构化输出的场景中，每一个信息节点都需要被"编排"进输出序列。

### 3.3 修正的特征语义描述

```
旧描述（05-13）:
  "多步结构化输出编排"
  → 听起来像一个"过程"
  
新描述（05-14, 基于 token 级激活）:
  "输出编排的全局增强信号"
  → 实际上是所有内容词上的均匀激活增强
  
更精确的说法:
  这些特征编码的不是"编排步骤"，而是
  "将这些 tokens 组织成结构化输出"的全局上下文信号。
  它们在需要多步推导的场景中增强所有内容词的表示，
  而非在特定逻辑词上爆发。
```

### 3.4 方法学启示

1. **Per-token tracing 是理解特征语义的必要补充**：仅凭因果干预（抑制后崩溃）不足以推断特征功能，需要结合激活模式分析
2. **单一 token 位置分析（last token）会错过全局模式**：传统的 `/sae` 只分析最后一个 token，无法发现这种跨所有位置的均匀激活
3. **简单 baseline（knowledge recall）是有效的负对照**：Capital 的弱激活（n_active=3-7 vs 13-17）量化了"需要结构化输出的程度"
